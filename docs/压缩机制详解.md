# 我们如何给 AI 编程助手做「上下文压缩」：别让它一压缩就失忆

做一个能长时间改代码的 AI 助手，你会很快撞上同一个墙：

**上下文窗口再大，也会被工具输出、日志、搜索结果、来回确认撑爆。**

窗口满了怎么办？最朴素的办法是「砍掉旧对话，留一句摘要」。听起来合理，实际一砍，模型经常变成这样：

- 忘了你让它改哪个文件  
- 忘了刚才测试已经挂了  
- 重新把整个仓库搜一遍  
- 甚至礼貌地问你：「你刚才想做什么？」

压缩不难，**压缩后还能接着干活**才难。

下面把我们在 iceCoder 里做的一整套「分层压缩 + 结构化恢复」讲清楚。尽量用普通人能听懂的话，适合做 Agent / 长会话产品的同学对照。

---

## 一、先把问题说成人话

可以把大模型的上下文，想成一张**办公桌**：

| 桌上的东西 | 现实对应 | 问题 |
|------------|----------|------|
| 刚打开的源码 | `read_file` 结果 | 占地方，但干活离不开 |
| 搜索结果、命令日志 | `grep` / `run_command` 输出 | 很快过时，却特别占空间 |
| 用户一句「继续」「跑测试」 | 短指令 | 字少，但丢了就断档 |
| 系统提醒、旧摘要 | 各种注入块 | 重复堆叠，越压越乱 |

桌子满了，你不能把整张桌掀翻，只留一张便利贴写「之前在改登录」。  
你得：**清掉垃圾、收起旧草稿、把关键文件和当前任务目标钉在眼前**。

这就是压缩机制要做的事。

---

## 二、核心原则：压缩不是「删历史」，是「换一种记得」

我们坚持几条：

1. **先轻后重** — 能本地清掉的，绝不先花一次大模型去「总结人生」  
2. **源码优先** — 读过、改过的文件，比搜索流水账更值钱  
3. **结构化状态 > 自然语言回忆** — 「目标 / 阶段 / 改了哪些文件 / 验证过没」写成机器可读状态，比让模型自己猜靠谱  
4. **压缩前先记账** — 先把会话笔记刷新，再砍历史，避免读到过期草稿  
5. **压缩后要会「续命」** — 重新注入任务状态、最近文件、恢复指引；必要时还能检测「失忆话术」并自动补救  

一句话：**省 token 是手段，任务不断档才是目的。**

---

## 三、什么时候开始压缩？两道水位线

我们不靠「消息条数到了 40 条就砍」这种粗规则当主开关，而是看**有效占用**相对上下文窗口的比例。

占用怎么算？取两者较大值：

- 本地估算：当前消息 + 工具 schema 大概占多少 token  
- 上一轮 API 回报的 `prompt_tokens`（更接近真实账单）

然后划两道线（相对当前模型窗口）：

```text
约 72%  ── 微压缩（轻量、本地、尽量对模型透明）
约 85%  ── 硬压缩（真正「搬家」：旧历史变摘要，只留最近一段）
剩余 < 18K token ── 也视为触线，直接硬压缩（给下一轮留呼吸空间）
约 93%  ── 主动激进收缩（应急 fork，防止顶破天花板）
```

直觉上：

- **72%**：房间有点挤了，先把过期报纸扔掉  
- **85%**：该大扫除了，旧对话收进「会话笔记 + 摘要」  
- **93%**：快爆仓了，强制腾地方  

微压缩还有配额：每轮最多 1 次，单会话累计有上限，避免「一直微压却压不动」空转。  
如果微压缩省下来的量太少（比如不到占用的 5%），同轮会**升档硬压缩**——别假装清干净了。

---

## 四、两道防线：微压缩 vs 硬压缩

### 4.1 微压缩：清垃圾，不掀桌子

触发后做的事很克制，而且**不调大模型**：

- **不删**用户的短句（「继续」「跑测试」「用 Vitest」这种导航/确认句，字少但关键）  
- 对白名单里的「过时工具结果」，把正文换成一行占位，例如：  
  `[Old tool result cleared for context] tool=grep (~12000 chars removed)`  
- 最近几轮（大约最近 5 次带工具调用的助手轮）的结果先留着  
- **文件读写类工具结果不清空**（源码是编码 Agent 的核心上下文）

可以清的典型工具：搜索、列目录、跑命令、网页抓取、子代理探索摘要等。  
不清的：`read_file` / `write_file` / `edit_file` 等文件操作。

微压缩理想状态是：**模型几乎感觉不到发生了什么**，只是上下文变轻了。

### 4.2 硬压缩：真的搬家

硬压缩才会动「对话骨架」：旧消息被摘要替代，只保留最近一段「还在干活」的后缀。

这里有两条路径：

| 路径 | 什么时候走 | 成本 | 摘要从哪来 |
|------|------------|------|------------|
| 会话记忆路径 | 磁盘上已有可用的会话笔记 | **0 次额外 LLM** | 直接用会话笔记当权威摘要 |
| LLM 路径 | 笔记不可用 | 通常 1 次 LLM | 本地结构化摘要，可选再让模型精炼 |

优先走会话记忆路径。原因很简单：长任务里，会话笔记本来就在后台维护；压缩时再花一次钱「总结历史」，又贵又容易和笔记打架。

---

## 五、硬压缩怎么「层层剥洋葱」

硬压缩不是一刀切，而是五层递进（能早停就早停）：

```text
1. snip          剪掉重复的系统提醒、旧摘要、旧压缩边界……
2. microcompact  旧工具调用：参数清空，结果缩成「成功/失败 + 短预览」
3. trim          超长工具输出截断（文件类限额更高）
4. structural    从即将丢掉的消息里，本地抽出结构化摘要（不调 LLM）
5. summarize     有会话笔记就用笔记；否则可选 LLM 精炼
```

### 第 1 层：snip —— 先删「重复便利贴」

历史里常堆着多份：

- `<system-reminder>`  
- `<context-summary>`  
- `<compact_boundary>`  
- `<recent-dialogue-focus>`  

同类只留最新一份，空的 assistant 气泡也丢掉。  
这一步便宜、确定、可测。

### 第 2 层：microcompact —— 旧工具调用「只留名片」

对**非最近**的工具交互：

- assistant 的工具参数清空（知道调过什么工具就够了）  
- tool 结果缩成状态 + 短预览  
- **文件操作结果尽量完整保留**（否则模型会反复重新读文件，更费）

### 第 3 层：trim —— 再长也得砍一刀

普通工具结果有字符上限；文件类上限更高。  
超长就截断并标注原始长度，避免单次 `cat` 一个巨文件把窗口吃光。

### 第 4～5 层：摘要

- 先从被丢弃的消息里**本地**抽出：改过什么、跑过什么、错过什么  
- 若有会话笔记：笔记直接成为 `<context-summary>` 的正文  
- 若没有：才考虑再调一次 LLM 把摘要写顺  

压缩后的消息大致长这样：

```text
[system 提示]
[context-summary：会话笔记或结构化摘要]
[compact_boundary：告诉模型「摘要之后才是可信后缀」]
[recent-dialogue-focus：压缩前最近两轮用户意图 + 助手状态]
[最近保留的真实对话后缀]
```

「边界块」很重要：它明确告诉模型——**摘要是背景，边界后面的消息才是接着干的锚点**，别回头去回答旧问题。

---

## 六、压缩前：先记账，再搬家

硬压缩前有一步很多人忽略、但特别关键：

**强制刷新会话笔记（带超时降级）。**

会话笔记是一份结构化 Markdown「今天的草稿纸」，大致包括：

- 会话标题 / 当前状态 / 任务规格  
- 关键文件与函数  
- 工作流与命令  
- 错误与纠正  
- 经验与关键结果  
- 运行时证据（机器维护的结构化快照）  
- 极短工作日志  

更新节奏平时偏懒（上下文涨一截、工具攒够几次、自然断点才写）。  
**但硬压缩前会 force 一次**：尽量把「此刻在干什么」写进笔记，再读盘拿来当摘要。

如果刷新超时，就用磁盘上已有内容继续压——宁可笔记略旧，也不要卡死整轮。

同时还会提前抠出几样「压缩后会丢、但恢复时要用」的东西：

- 最近注入的长期记忆提醒（避免偏好丢光）  
- 最近读过的文件内容（准备重注入）  
- 当前 `TaskState` + `RepoContext` 快照（准备做成运行时恢复块）

---

## 七、压缩后：五件「续命」装备

砍完历史，不是结束，而是开始恢复。我们按优先级往回塞：

### 1）会话笔记摘要（若走了记忆路径）

告诉模型：当前会话状态以笔记为准；  
优先级是：**当前对话 > 会话笔记 > 长期记忆**。  
有冲突就信会话笔记，重要矛盾可以跟用户说一声。

### 2）Runtime Recovery Context（结构化任务锚）

这是硬压缩后的「权威便利贴」，大致写明：

- 目标 goal、意图 intent、阶段 phase  
- 要不要验证、验证过没  
- 建议的下一步 nextAction  
- 改过的文件、读过的文件  
- 最近命令 / 测试命令 / 诊断信息  

它有 token 预算；超预算会按规则逐步瘦身（先砍次要已读文件列表，再砍命令与诊断，目标与关键改动尽量留到最后）。

原则：**结构化状态优先于「模型自己回忆」**。

### 3）最近文件内容重注入

从压缩前历史里捞最近读过的源码（路径去重、条数动态上限、总 token 预算封顶），再塞回去。  
顺带可带上最近几次搜索结果的短摘录，减少「失忆后重新全库乱搜」。

### 4）恢复指引（对人话版操作手册）

明确要求模型：

- 直接从断点继续  
- 不要复述「我理解上下文被压缩了」  
- 不要让用户把需求再说一遍（除非笔记和上下文里真的都没有任务）  
- 旧消息里的问题不要回头回答  

有会话笔记时，还会强调：笔记里除了人读段落，还有机器可读的 `icecoder-runtime` JSON 块，重启后续聊也能对齐任务/仓库状态。

### 5）失忆检测兜底

即使上面都做了，模型偶尔仍会说：「抱歉，我不太清楚你之前的任务……」  
我们会在「刚压缩完」的窗口里识别这类话术，**自动再补一轮任务上下文**（每轮压缩后限次），避免礼貌性失忆把长任务打断。

另外，纠偏类、接管类、关键恢复注入可以打上 `preserveOnCompaction`，硬压缩切后缀时尽量保住——别把「系统刚纠正过的红线」也一起扔掉。

## 八、为什么还要子代理？因为压缩治标，膨胀治本

压缩是「桌子满了再收拾」。更好的办法是：**别把整袋垃圾倒桌上。**

探索代码库时，主会话如果直接塞满搜索命中和文件全文，会加速触线。  
我们用只读子代理：它在隔离循环里搜、读，最后只回主会话一份**短摘要**（大约几百 token 量级）。

经验上，探索引起的上下文膨胀能少一大截。  
压缩仍然需要，但触发会晚很多，恢复压力也小很多。

## 九、一张图串起来

```text
每轮准备上下文
  │
  ├─ 占用 < 72% ─────────────────────────── 什么都不做
  │
  ├─ 72% ≤ 占用 < 硬压缩线
  │     └─ 微压缩：清空过时工具正文（本地）
  │           ├─ 省得够 → 结束
  │           └─ 省太少 / 仍触线 → 升档硬压缩
  │
  └─ 硬压缩线 / 剩余不足 / 升档
        ├─ 强制更新会话笔记（超时则用旧笔记）
        ├─ 提取：记忆提醒 / 文件内容 / Runtime 快照
        ├─ snip → microcompact → trim → 摘要
        │     ├─ 有笔记 → 0 LLM，笔记当摘要
        │     └─ 无笔记 → 结构化摘要（可选 LLM）
        ├─ 注入：边界块 + 近两轮对话焦点 + 最近后缀
        └─ 恢复：Runtime 状态 → 文件内容 → 恢复指引
              └─ 若仍「失忆话术」→ 自动补救一次
```

## 十、设计上我们坚持的几条（可直接抄）

1. **水位线用占用比例，不只用消息条数** — 一条巨长工具输出，比二十句闲聊更危险。  
2. **双轨占用** — 本地估算和 API 回报取大，少漏判。  
3. **微压缩零 LLM、硬压缩优先零 LLM** — 会话笔记是压缩的「预付摘要」。  
4. **文件操作结果特殊对待** — 源码不是普通日志。  
5. **短指令别当垃圾删** — 微压缩阶段尤其小心。  
6. **压缩边界要显式** — 告诉模型哪里开始才是「现在」。  
7. **恢复要分层** — 笔记叙事 + 结构化 Runtime + 文件实体 + 操作指引 + 失忆兜底。  
8. **从源头减膨胀** — 子代理探索、工具结果裁剪，比只会压缩更重要。  
9. **可观测** — 记录压缩前后消息数/token、节省量、恢复是否成功，才能调参而不是玄学。  

---

## 十一、局限：我们清楚还不够好的地方

- **摘要质量仍依赖会话笔记是否及时** — 笔记刷新失败或超时，恢复会偏旧。  
- **重注入有预算** — 读过很多文件时，不可能全塞回去；只能保最近、保关键。  
- **模型仍可能忽略结构化锚** — 所以才有失忆检测，但它不是万能药。  
- **激进 fork 是最后手段** — 能保活，但对「完整叙事连贯性」有损，属于止损。  

这些限制换来的是：行为可测、成本可控、长任务不断档——对本地 / 自托管编程助手，往往比「永远不压缩、指望超大窗口」更现实。

---

## 十二、若你也要做 Agent 压缩，可直接对照的检查清单

- [ ] 触发条件是 token 占用，还是只数消息条数？  
- [ ] 有没有「轻量清理」层，避免动不动全量摘要？  
- [ ] 压缩前有没有强制落盘当前任务状态？  
- [ ] 摘要优先用会话笔记，还是每次都调 LLM？  
- [ ] 源码读写结果是否被当成普通日志一起砍掉？  
- [ ] 压缩后有没有结构化 Runtime（目标/文件/验证）而不只靠自然语言？  
- [ ] 有没有明确告诉模型「摘要之后才是可信后缀」？  
- [ ] 短指令、纠偏注入会不会被误删？  
- [ ] 压缩后若模型开始「礼貌失忆」，有没有自动补救？  
- [ ] 探索类任务是否用子代理，避免主上下文被污染？  

---

## 结语

上下文压缩的本质，不是「把历史变短」，而是：

**在窗口不够用时，用最低成本丢掉可替代信息，并确保任务目标、关键文件与下一步动作，仍然钉在模型眼前。**

我们用「微压缩清垃圾 → 硬压缩搬家 → 会话笔记当预付摘要 → Runtime 结构化续命 → 失忆兜底」这条链路，是因为编程 Agent 的上下文里，真正贵的不是聊天语气，而是**可执行的工作状态**。

省 token 谁都会；  
**压完还能接着改、接着测、接着交付**，才是工程问题。

如果你也在做长会话 Agent，欢迎对照这套「双水位线 + 分层压缩 + 零成本笔记路径 + 结构化恢复」的拆法，按自己的产品形态裁剪。  
压缩可以很凶，但恢复必须很温柔——对用户任务温柔，对 token 账单也可以不温柔。
