# 1. 系统概述

iceCoder 双模自适应运行时监管系统，是位于 Harness 主循环之上的运行时控制层。

其目标是：

在最大限度保留大模型自由执行能力的同时，为关键任务提供强规则兜底与异常恢复能力。

系统职责：

- 自由执行
- 异常观察
- 风险识别
- 系统接管
- 状态恢复
- 安全交还

本系统本质不是 planner，也不是 workflow。

而是：

> AI Agent 运行时监管内核（Runtime Supervisor）。

# 2. 核心设计哲学

---

## 2.1 信任优先

默认信任模型执行能力。

系统不主动限制模型。

仅在必要时介入。

---

## 2.2 关键任务监管

仅对高风险关键任务启用强规则。

非关键任务完全自由。

---

## 2.3 状态优先恢复

恢复逻辑必须基于当前工作区真实状态。

不得仅依赖执行轨迹。

---

## 2.4 成本有界

所有恢复流程必须受预算控制。

防止无限修复循环。

---

## 2.5 全程可观测

所有运行事件必须记录。

用于 benchmark 与调优。

## V1 计算因子（启发式，加权求和 → 0~1）

| 因子 | 说明 | 方向 |
|------|------|------|
| Git 工作区 | 干净 / 仅预期文件变更 | 越高越可信 |

# 8.10 恢复监管编排器（RecoverySupervisor）

**实现规范核心。** 编排 §8.1–§8.9，对外暴露单一入口；Harness 四钩子只调本模块（经 bridge）。

## 职责

- 每轮/每工具后：汇总 `PassiveObserver` → 结合 `ModeController` 与 §9 三条件 → 输出 **`SupervisorDecision`**
- 触发接管：走 **§10 恢复主路径**（反构图成功）或落入 **§19.2 降级旁路**
- 接管期间：**唯一**向 `msgs` 注入纠偏类 system/user 块（与 §19.6 互斥表一致）
- 协调 `GraphExecutor`（`replaceGraph`、`setEvaluationMode`）、`RecoveryBudgetManager`、`StabilityWindow`

## 不负责

- 不替代 `GraphExecutor` 的节点合约执行（仍由 TaskGraph 负责）
- 不直接执行 rollback（经工具 + `confirm`，见 §19.5）
- **`RecoveryBoundary`**（原 §11 提及）职责并入本模块：**接管期间**禁止其他 Resilience 模块向 `msgs` inject（见 §19.6）

## 核心 API（附录 A 全文）

```ts
interface RecoverySupervisor {
  /** 每轮工具后或 before reply 调用 */
  evaluate(ctx: SupervisorEvaluateContext): Promise<SupervisorDecision>;
  /** 进入 supervised：写 timeline、注入接管块、换图或强提示 */
  applyTakeover(ctx: TakeoverContext): Promise<void>;
  /** 稳定窗口通过后交还 */
  applyHandoff(ctx: HandoffContext): Promise<void>;
}

type SupervisorDecision =
  | { action: 'continue' }
  | { action: 'takeover'; reason: string; signals: DeviationSignal[] }
  | { action: 'handoff_pending' }
  | { action: 'handoff' }
  | { action: 'fail'; kind: 'checkpoint' | 'rollback' };
```

## 与 ModeController 分工

| 组件 | 职责 |
|------|------|
| `ModeController` | 解析 `ICE_SUPERVISOR_MODE`；提供 §17 参数；维护 `supervisorPhase` 与冷却计数 |
| `RecoverySupervisor` | 根据观测与阈值调用 `evaluate`；驱动 §10 / §19.2 / §12 |

## 14.1 钩子位置

| 钩子 | 插入时机（约） | 职责 |
|------|----------------|------|
| before tool call | 工具执行前 | supervised 下合约门禁；`GraphExecutor.checkToolCall` |

# 15. 配置项

## 15.1 运行模式

环境变量：**`ICE_SUPERVISOR_MODE`**（建议；实现时写入 `data/supervisor-config.json` 亦可热加载）

| 值 | 模式 | 行为摘要 |
|---|---|---|
| `off` | 自由 | 关闭 Supervisor（零观测开销）；等同今日纯 Harness |
| `adaptive` | 双模（**默认**） | 平时自由；满足 §9 三条件时接管 |
| `strict` | 全图 | 关键域任务 **首轮即建图**，全程图驱动（复用现有 TaskGraph） |
| `supervised` | 高监管 | 平时自由；**更低阈值、更短恢复预算** 的 adaptive 预设（见下） |

**`adaptive` 与 `supervised` 关系：**

- 二者共用 **同一套** `RecoverySupervisor` + `GraphExecutor`，不是两套引擎。
- **`supervised`** = 配置预设：更低风险阈（0.4）、更短恢复预算（§17 `supervised` 列）。
- **`adaptive`** = 默认预设：自由段用 0.6；**一旦接管**，参数使用 §17 **`adaptive·接管段`** 列（与 `supervised` 列相同，见 §17 注释）。

**与 `ICE_TASK_GRAPH`：** 以 `ICE_SUPERVISOR_MODE` 为准；`ICE_TASK_GRAPH` 标记为遗留，实现后不再读取。

## 15.2 Shadow 与 A/B

**前置：** `ICE_SUPERVISOR_MODE` 不得为 `off`（`off` 不启动 Observer）。评测时建议 `ICE_EVAL_MODE=1` 配合 `supervisor=off`，或仅 `shadow` 记日志。

| 行为 | Shadow ON | Shadow OFF |
|------|-----------|------------|
| `PassiveObserver` / `RiskEvaluator` | 正常运行 | 正常运行 |
| `RecoverySupervisor.evaluate` | **完整执行** | 完整执行 |
| `supervisorPhase` / 真实接管 | **禁止**（`takeover` / `handoff` 不得改 phase） | 按 decision 执行 |
| `EventTimeline` | **必须写入**（含「本会接管」原因） | 正常写入 |
| 向 `msgs` inject | 仅 shadow 诊断块（可选），无接管块 | 按 §16.1 / §19.6 |

- **`ICE_SUPERVISOR_SHADOW=1`**：用于 benchmark 对照与阈值调参；与 **§22** 指标 `switch frequency` 等对齐。
- `off` 模式下不启动 `PassiveObserver`（零成本路径）。

# 16. 实现决议（冻结 baseline）

以下决议为编码前基线；变更需更新本节并注明版本。

| 决策 | 决议 | 理由 |
|------|------|------|
| 接管时是否清空对话？ | **保留历史 + 注入接管块** | 纠正不是重启；`TaskState` / `RepoContext` / checkpoint 依赖连续上下文 |
| strict vs supervised | **同架构、不同参数** | 复用 `GraphExecutor` + `RecoverySupervisor`；仅阈值与「何时建图」不同 |
| GoalDrift | **V1 启发式 → V2 LLM** | 零成本覆盖大部分偏离；V2 仅灰区再调 LLM |
| 反构图失败降级 | **三级：模板图 → 强提示 → 人工** | 任何情况有明确兜底 |
| 评估去重 | **接管后 `evaluateRound` 仅记录 metrics** | Supervisor 链为唯一纠偏出口，避免双份 `inject_hint` |
| rollback | **经工具调用 + `confirm` 权限** | 统一权限门禁，保护用户数据 |

### 16.1 接管块格式

使用固定结构化块（建议 fence 语言标签 `icecoder-supervisor-takeover`），**压缩时不可 snip**：

- 用户 `goal`
- 触发原因（异常信号摘要）
- `WorkspaceSnapshotRef` 摘要（非全量快照，见 §14.2）
- 当前 `supervisorPhase`、恢复图首节点 ID（若有）

### 16.2 strict 与 supervised 的产品语义

| | strict | supervised |
|---|--------|------------|
| 用户感知 | 全程有任务图路线 | 平时像 Copilot，卡住时系统接管 |
| 建图时机 | 关键域 **第 1 轮** `initGraph` | **异常触发后** `replaceGraph` / 模板图 |
| 引擎 | 同一套 `GraphExecutor` + `RecoverySupervisor` | 同左 |

# 17. 模式参数矩阵

`strict`、`supervised`、`adaptive` **共用同一实现**；下表为默认参数（可进 `supervisor-config.json` / **附录 A** `SupervisorConfig`）。

> **`adaptive·接管段`**：`ICE_SUPERVISOR_MODE=adaptive` 且 `supervisorPhase=supervised` 时使用本列（与 `supervised` 模式列一致）。自由段参数勿用于接管后。

| 参数 | strict | supervised（模式） | adaptive·自由段 | **adaptive·接管段** |
|------|--------|------------------|-----------------|---------------------|
| 首轮是否建图（关键域） | 是 | 否 | 否 | 否（接管时 `replaceGraph`） |
| 风险阈值（进入监管候选） | 0.5 | 0.4 | 0.6 | —（已接管，不重复评估） |
| 最大恢复轮数 | 5 | 3 | — | **3** |
| 恢复 token 占总预算 | 30% | 25% | — | **25%** |
| 稳定窗口观察轮数 | 2 | 3 | — | **3** |
| handoff 后冷却轮数 | 2 | 3 | — | **3** |
| `evaluateRound` 注入纠偏 | 全程 `full` | 接管后 `metrics_only` | 无图 / 仅记录 | **`metrics_only`** |
| `checkToolCall` | 全程 block/warn | 接管后 block/warn | 无图时不检查 | **block/warn** |

**说明：** `ICE_SUPERVISOR_MODE=supervised` 时，全程按 `supervised（模式）` 列 + 接管段列；从未建图直到首次 takeover，接管段列生效。

# 18. Supervisor 运行时状态机

```text
                    ┌─────────────┐
         启动 ──────►│    free     │◄────────────────┐
                    └──────┬──────┘                 │
                           │ §9 三条件满足            │ handoff 通过
                           ▼                         │ + 冷却结束
                    ┌─────────────┐                 │
                    │ supervised  │─────────────────┤
                    └──────┬──────┘                 │
                           │ 校准完成               │
                           ▼                         │
                    ┌─────────────┐    失败         │
                    │handoff_pending├──────────────►│ cooldown
                    └──────┬──────┘                 │
                           │ 稳定窗口 OK（轮数见 §17）│
                           └────────────────────────┘

失败出口：RecoveryBudget 耗尽 / safety 失败 → §19.2 三级 · 人工 或 §19.5 rollback
```

| 阶段 | 含义 |
|------|------|
| `free` | 模型主导；`PassiveObserver` 仅观测（或 shadow 只记日志） |
| `supervised` | 图驱动或 Supervisor 强提示主导；`evaluateRound` 不 inject |
| `handoff_pending` | 稳定窗口计数中，尚未交还 |
| `cooldown` | 交还后 N 轮内禁止再次接管 |

# 19. 关键机制细化

## 19.1 GoalDrift（V1 启发式）

V1 作为 `PassiveObserver` 上的合成信号，**不单独再开一套检测**，与现有 Harness 信号合并：

- 目标关键词 vs 本轮工具类型（长期只读/搜索 → 漂移）
- `filesChanged` 为空但 phase 已进入 editing
- 用户 goal 与最近 assistant 内容的 bigram Jaccard（可复用任务切换 Jaccard）
- 连续 N 轮 `TaskState.phase` 无推进

**V2 LLM：** 仅当 V1 分数处于灰区（建议 0.35–0.55）且已在监管候选域时触发 side call。

默认阈值：alignment **< 0.45** 连续 **2** 轮（V1 启发式分数；V2 见 §8.3）。

## 19.2 反构图失败：降级旁路（与 §10 区分）

**不重复执行 §10 全流程。** 仅在「需要接管但模板图不可信/构建失败」时进入。

```text
§10 恢复主路径（builder 成功）
        │
        ├─ 成功 → supervised 执行
        │
        └─ 失败 ↓
§19.2 三级降级
   1 模板图（重试 builder / 降置信要求）
   2 强提示（无图，Supervisor 单条 inject）
   3 人工（user_checkpoint）
```

| 级 | 条件 | 行为 |
|----|------|------|
| **1 模板图** | `SnapshotConfidence ≥ 0.65` 且 `RecoverySafetyChecker.ok` | `RetrospectiveGraphBuilder` 按 intent 模板建图，标记已完成节点 |
| **2 强提示** | 1 失败或预算内重试 | 不建图 / `markGraphPaused`；**仅** `RecoverySupervisor` 注入 `[System Recovery]`；**本轮禁止再次 takeover** |
| **3 人工** | 2 仍失败或预算耗尽 | `stopReason: user_checkpoint`；`EventTimeline` 记 `failure`；UI 提示 |

### 模板图最小集（V1）

| intent | 来源 | 已完成节点判定 |
|--------|------|----------------|
| `debug` / `edit` / `test` / `refactor` | 复用 `buildGraph(goal, intent)`（`task-graph-builder.ts`） | `TaskState.filesChanged` 非空 → 对应 inspect 标 `done`；`verificationStatus=passed` → verify 标 `done` |
| fallback | 现有 `fallbackBranches` 机制 | 与 TaskGraph V1 一致 |

不在 V1 做全 LLM 重规划；仅允许 LLM 生成剩余步骤**标题**（可选，受 `RecoveryBudget` 约束）。

## 19.3 `GraphExecutor` 扩展（接管必备）

- `loadGraph` / `replaceGraph`：中途换图（supervised / 恢复）
- `setEvaluationMode('full' | 'metrics_only')`：接管后纠偏去重（§16 表）
- `enterSupervisedMode` / `exitSupervisedMode`：与 `supervisorPhase` 同步

## 19.4 评估去重（接管后）

- **接管前（free）：** 无图 → 不 `evaluateRound`；strict 全程有图 → `full`。
- **接管后（supervised）：** `evaluateRound` **只写 metrics / checkpoint**，不向 `msgs` 注入 hint。
- **`checkToolCall`：** 接管后仍可做 block/warn（执行前门禁，与轮末 inject 无关）。

## 19.5 rollback

- 仅通过 **builtin 工具**（如 `run_command` / 专用 `rollback_workspace`）执行。
- 走 Harness **`confirm`** 权限；文案须包含将执行的命令与影响文件列表（来自 snapshot）。
- 用户拒绝 confirm → 进入 **三级降级 · 人工**，不静默失败。
- rollback 失败计入 `RecoveryBudgetManager`。

## 19.6 与现有 Resilience 互斥（`supervisorPhase === supervised`）

接管后 **仅 `RecoverySupervisor` 可向 `msgs` 注入纠偏类内容**；其余模块按下表降级。

| 信号源 | free / shadow | supervised |
|--------|---------------|------------|
| `consecutiveToolFailures` 长文案 inject | 启用 | **关闭**（已计入 Observer） |
| `resilienceMaybeBranchRecover` | 启用 | **关闭**（branch budget 仍计数） |
| `resilienceMaybeReviewStep` inject | 启用 | **关闭**（review 结果仅记入 timeline） |
| `consecutiveReadOnlyRounds` 提示 | 启用 | **关闭**或合并为 Observer 一条 |
| `GraphExecutor.evaluateRound` inject | strict: full；free: 无 | **`metrics_only` 仅** |
| `GraphExecutor.checkToolCall` block/warn | 按 §17 | **启用**（执行前门禁） |
| `RecoverySupervisor` 接管块 / Recovery | 否 | **唯一 inject 源** |

# 23. 对比定位

与 Claude Code / Codex CLI / Devin 等相比：

| 维度 | 典型工具型 Agent | iceCoder V1.3 Supervisor |
