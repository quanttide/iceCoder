## 1. 审计结论摘要

原方案文档头部仍标注「待评审（未实施）」，与当前代码**部分脱节**：Phase 1 核心与 Phase 2/3 部分能力**已落地**，但召回 LLM 门槛、Dream 空转、索引 bypass、遥测缺口等问题**仍未解决**。

| 维度 | 结论 |
|------|------|
| 索引写时维护 | Extract / Dream 删盘已接 `memory-index-maintainer`；`MEMORY.md` 缺失时仍跳过 upsert |
| Extract 去重 | **已 hardcode `merge` @ 0.85**，非 shadow；不存在 `ICE_EXTRACT_DEDUP` |
| Dream 规则合并 | **`ICE_RULE_MERGE` 默认 `shadow`**，候选对只打日志不合并 |
| Recall LLM | 结构性门槛导致 `usedLLM` 极低；Harness 精排不计入该字段 |
| Dream no-op | 历史遥测 ~59% `executed=false`；锁竞争 + LLM 空 JSON 仍常见 |
| 写入质量门控 | 仅有 `confidence ≥ 0.6` 等规则过滤，**无独立「信息密度分数」二次粗筛** |

扩展项见 **§9**（召回双精排、Web 删盘索引、远程配置死字段、LoCoMo 未落地步骤等 30+ 项）。

## 2. 六项缺口核查（修正版）

以下对照常见误判清单，给出**代码级**结论。

| # | 原表述 | 是否存在 | 修正说明 |
|---|--------|:--------:|----------|
| 1 | 没有「二次粗筛」/ 信息密度分数，写回全靠 LLM 自觉 | ⚠️ 部分 | 有 `meetsExtractionConfidenceThreshold`（≥0.6）、category/filename 门控；**无**独立密度评分或写盘前二次程序化打分 |
| 2 | `ICE_EXTRACT_DEDUP` 默认 `shadow`，去重空转 | ❌ 不准确 | **该环境变量不存在**；Extract 在 `memory-llm-extractor.ts` **写死 `'merge'`**。默认 shadow 的是 **`ICE_RULE_MERGE`**（Dream 后规则合并） |
| 3 | Recall 端 LLM 精排几乎未启用 | ✅ 是 | `LLM_RECALL_MIN_CANDIDATES=4`；Harness `rerankMemories` 需候选 > `2×finalK`（默认约 80）；`usedLLM` 不含 Harness 精排 |
| 4 | Dream ~60% no-op | ✅ 历史成立 | 91 次遥测中 54 次 `executed=false`；门控 `shouldRun=false`（如 `index_drift_rule_rebuild`）**不入队**，不计入该分母 |
| 5 | 索引漂移（如 MEMORY.md 13 条 vs 磁盘 91 文件） | ⚠️ 部分缓解 | `upsertIndexRow` 在 MEMORY.md **不存在或为空时直接 return**；`write_file` 直写、`FileMemoryManager.saveMemory`、外部脚本均 bypass |
| 6 | 无低门槛重复合并（0.85 + shadow） | ⚠️ 部分 | Extract merge@0.85 **已生效**；Dream 规则合并 shadow@0.88 **不执行**；无文件名前缀聚类等更低门槛规则 |

### 2.1 易混淆：两套去重机制

```text
Extract 写时去重（memory-llm-extractor.ts）
  → checkExtractDedupSync(..., 0.85, 'merge')   ← 硬编码 merge，无 env

Dream 后规则合并（memory-dream.ts）
  → findMergeCandidates(..., 0.88)
  → performRuleMerge(..., ICE_RULE_MERGE || 'shadow')   ← 默认只日志
```

方案 §6 中 `dedup.mode: "shadow"` 的 JSON 示例**尚未接入** `DedupConfig` 运行时；勿与 Extract 去重混谈。
