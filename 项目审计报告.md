## 一、安全相关

> 说明：Shell / 文件工具的「沙箱与工作区边界」为有意设计（按本机工具定位放开），不在本报告范围，相关条目已移除。

### ✅ P0-3【已修复】软超时 adopt 失败后子进程无硬超时（进程泄漏）

`src/tools/builtin/shell-tool.ts:249,267-270`

软超时触发时先清掉 `hardTimer`，若随后 `adopt()` 失败（如并发上限），代码设 `escalated = false` 直接 return，**没有重新装回硬超时**，同时已 `removeAllListeners`（`shell-tool.ts:252-255`）。结果：子进程可能无限运行且无人收集输出。

> **修复（2026-06-26）**：adopt 失败分支改为立即 `SIGTERM` + 2s 后 `SIGKILL`（`unref` 不阻塞退出）终止子进程，并用已捕获的部分输出 `safeResolve` 让 Promise 落定，杜绝孤儿进程与悬挂 Promise。

### ✅ P0-4【已修复】MCP 启动失败残留僵尸进程

`src/mcp/mcp-manager.ts:106-131`

`startServer()` 的 catch 仅把状态置为 `error`，**未调用 `client.stop()`**。若 `spawn` 成功但 `initialize()`/`listTools()` 失败，子进程会一直挂着。`mcp-client.ts:104-106` 的 `process.on('error')` 也只打日志，不会 reject 等待中的 `start()`。

> **修复（2026-06-26）**：catch 中增加 `await client.stop()`（对未 spawn 的情况是安全空操作）回收子进程；stop 自身异常单独 `console.error`，不影响错误状态登记。回归测试见 `test/mcp/mcp-manager-startup-cleanup.test.ts`。

### 🟠 P1-5 全部 REST API 与 PC 端 WebSocket 无鉴权

`src/web/setup-gate.ts:18-32` 只在「未完成初始配置」时拦截 `/api/*`（放行 `/api/config`），这是 setup gate 而非鉴权。配置完成后以下接口对**同一局域网内任何人**开放：

| 接口 | 文件 |
|------|------|
| `POST /api/tools/execute`（直接执行工具） | `src/web/routes/tools.ts:38-69` |
| `/api/sessions/*`（读写删会话、图片、历史） | `src/web/routes/sessions.ts` |
| `POST /api/chat/upload` | `src/web/routes/upload.ts:139` |
| `POST /api/remote/session`（铸造远程 token / 使旧会话失效） | `src/web/routes/remote.ts:119-165` |
| PC 端 WebSocket（无 token） | `src/web/chat-ws.ts:1086-1087` |

移动端 upgrade 有 token 校验（`chat-ws.ts:1074-1085`），但 REST 仍然全开。配合 `skipPermissionChecks: true`（`chat-ws.ts:1597,1645`）在共享网络上风险更高。

**建议**：若定位为「纯本机工具」，默认绑定 `127.0.0.1`；若需局域网访问，给 REST + PC WS 加统一 token。

### 🟠 P1-6 其它输入未校验直达 `exec` / `fs`

- `src/tools/builtin/git-tool.ts:27`：`exec(`git ${args}`)` 字符串拼接，Windows 下经 cmd 执行，`args` 含 `;`/`&&` 可命令注入。

- `src/web/routes/sessions.ts:424-430`：`PUT /:id` 把任意 JSON 写入会话文件，无 schema 校验。

### 🟠 P1-7 仓库内疑似敏感/无关文件

- `cookies.txt`（已被 git 跟踪）：当前为空壳（仅 Netscape header），但**不应进版本库**，且易被误填真实 cookie。✅

## 二、稳定性与并发

### 🟠 P1-8 聊天运行的 sessionId 与 WS 订阅不一致 ✅

`src/web/chat-ws.ts:1371` vs `1446`

调用方用 **每连接订阅** 计算会话：
```ts
const runSid = wsToSubscribedSession.get(ws) || activeSessionId;  // 1371
```
但 `handleChatMessage` 内部却用 **全局** 活动会话：
```ts
const runSessionId = activeSessionId;  // 1446
```
当某个 WS 订阅的会话 ≠ 全局 `activeSessionId` 时，batch/状态广播用 `runSid`，而真正的运行、持久化、工具记录用 `activeSessionId` —— **写入到了错误的会话**。注释声称「锁定本次运行 sessionId」，但与调用方的 per-WS 模型自相矛盾。

### 🟠 P1-9 处理中标记与中止控制器是「连接级/全局」而非「会话级」

- `isProcessing` 是单连接变量（`chat-ws.ts:1145-1147`）：多标签页可对**同一会话**并发跑两个 harness。
- `activeAbortController` 是全局单例（`chat-ws.ts:330,1591-1592`）：一次 stop 会中止「碰巧设置了它」的那个运行，造成跨标签/跨会话误中止。

### 🟠 P1-10 进程级可变单例在并发会话下会串话（harness）

- `src/harness/intent-checkpoint-turn-snapshot.ts:30-31,46`：单个 `activeTurn` + `turnBuffers`，由 `chat-ws.ts` 触发；第二个并发 turn 会覆盖 `activeTurn`，导致预写快照挂到错误的消息/会话。
- `src/harness/verification-exempt-config.ts:24-25`（在 `harness.ts:472` 每次运行被设置）：后一次运行覆盖前一会话的验收豁免配置。
- `src/harness/sub-agent-runner.ts:81`：模块级 `subAgentCache` 跨所有会话共享。

> 这些在「同一进程内同时跑多个会话」时才暴露，但 Web 端多标签 + 移动端并发完全可能触发。

### 🟠 P1-11 会话删除时大量运行期资源未清理

`purgeSessionRuntimeCaches`（`chat-ws.ts:305-318`）清理了一部分，但以下未清，导致内存增长/后台进程残留：

- `runningTurns`（`chat-ws.ts:672`）——陈旧快照
- `pendingConfirms` 及其 60s 定时器（`chat-ws.ts:342-360,1661-1672`）
- `background-task-manager.ts:918` 的 `managersBySession`——**后台进程继续运行**
- `upload.ts:45` 的 `uploadedFiles`（无清理钩子）

### 🟠 P1-12 上传文件 Map 无界增长 + 临时文件不清理

`src/web/routes/upload.ts:45,164`：`uploadedFiles` Map 永不裁剪；`os.tmpdir()/iceCoder-uploads` 下的临时文件用后不删。上传错误还以 HTTP 200 返回 `{ error }`（`upload.ts:150,158,178`）。

### ✅ P0-13【已修复】流式重试会重复输出

`src/llm/llm-adapter.ts:159-169`

`withRetry` 包裹了整个 `stream()`：若流在已吐出若干 chunk 后失败，重试会重新 `provider.stream()`，回调收到**重复 delta**，无去重/重置保护。

> **修复（2026-06-26）**：`stream()` 用一个跟踪回调记录本次尝试是否已向调用方推送过任何内容（正文或 reasoning 均计入）；`withRetry` 新增可选 `canRetry` 守卫，**仅在尚未产出任何 chunk 时才允许重试**（覆盖"连接刚建立即失败"等典型可重试场景），产出后失败则直接上抛，杜绝重放重复。回归测试见 `test/llm/stream-retry-dedup.test.ts`。
>
> 关于"失败流不发 done"：经核实本仓库消费方（`harness-llm-call.ts:118`）依赖 Promise 的 resolve/reject 而非 `done=true` 判完成，失败时 Promise reject 即为正确信号，不会悬挂；若在失败路径补发 `done=true` 反而会触发一次伪"完成"step 事件，故不修改。

### 🟠 P1-14 记忆子系统无原子写 + 读改写竞态

- 全子系统**没有「写临时文件 + rename」**的原子写：`memory-index-maintainer.ts:186,195,205,269`、`memory-dream.ts:1276,1307,1367` 等。进程在写 `MEMORY.md`/dream 状态中途崩溃会损坏文件。
- `src/memory/file-memory/memory-recall.ts:1055-1093`：召回元数据按帧 flush 是读-改-写且无文件锁，会与 dream/extractor 改同一 frontmatter 竞争，错误还被 `.catch(() => {})` 吞掉（`1090-1092`）。
- 索引锁仅进程内（`memory-index-maintainer.ts:27`）：Web + CLI + dream 多进程并发可损坏 `MEMORY.md`。

### 🟠 P1-15 CLI 退出绕过优雅关闭

`src/cli/commands/chat.ts:201-207,489-495`：`/quit` 与 `rl.on('close')` 调用 `shutdown().catch(() => {})` 后直接 `process.exit(0)`，跳过 `drainMemory`、后台任务释放顺序与关闭超时——可能丢失未落盘的记忆/状态。`graceful-shutdown.ts:36-68` 本身设计良好，只是这些路径没用它。

## 七、建议的处理顺序

1. **进程/资源收口（P0）**：MCP 启动失败时 `client.stop()`；shell 软超时 adopt 失败时重装硬超时。
2. **会话隔离（P1）**：统一 `handleChatMessage` 用 `runSid` 而非全局 `activeSessionId`；把 `isProcessing`/`abortController` 改为按会话维护；会话删除时清理 `runningTurns`/`pendingConfirms`/后台任务/上传。
3. **可观测性（P1）**：把静默 `.catch(() => {})` 改为分级日志；持久化失败从 `console.debug` 升级为 `warn` 并上报。
4. **数据安全（P1）**：记忆/索引写入改为「临时文件 + rename」原子写；跨进程加文件锁。
5. **流式稳健（P0）**：流重试改为「仅在未产出任何 chunk 时重试」或加 delta 去重；失败路径也发 `done`。
6. **仓库卫生（P2/P3）**：删除 `cookies.txt`、`*.tgz` 等入库产物；统一为单一包管理器；统一 Node 版本声明；清理死代码与重复逻辑。
